The Marimba v1.2.0 release makes every packaged dataset richer and more FAIR, with stronger machine-readable metadata throughout: stable dataset and image identity, machine-readable provenance, embedded rights and attribution, and spatial-discovery metadata. The iFDO output is upgraded to schema v2.2.1, and packaging is more robust in restricted network environments.

## What's New in v1.2.0

- **Machine-Readable Provenance**: Every packaged dataset now includes a W3C PROV-O `provenance.json` record capturing how it was produced: the pipeline's git commit and repository URL, the Marimba and ExifTool/FFmpeg versions used, the dataset version, and the packaging time, alongside a deterministic image-curation-protocol summary. This makes a dataset's lineage auditable without re-running the pipeline.
- **Deterministic Dataset and Image Identity**: The `image-set-uuid` is now derived deterministically from the dataset name, and each `image-uuid` is derived from the set UUID and the image's dataset-relative path when a pipeline leaves it unset. Re-packaging the same input produces the same identifiers, so dataset identity is stable and reproducible rather than random per run.
- **Spatial Bounding Box**: The iFDO header now records the dataset's spatial extent (`image-set-min/max-latitude/longitude-degrees`), so datasets can be discovered and filtered by geographic area without opening every image record.
- **Discrete Rights and Attribution Metadata**: Creator names, ORCIDs, licence, copyright, and identity are now embedded as discrete, standard EXIF and XMP fields (creators to `Artist`/`dc:Creator`, ORCID to `plus:ImageCreator`, licence and copyright to the `xmpRights` block, and the image UUID to `dc:Identifier`), so attribution and licensing travel with each file in fields that standard tools understand.
- **GPS Completeness**: Geotagged imagery now carries UTC `GPSDateStamp`/`GPSTimeStamp` fix-time tags and a `GPSMapDatum` tag that honours the pipeline-supplied coordinate reference system, making the embedded positions self-describing and unambiguous.
- **Automatic Citations**: Each dataset's `image-set-related-material` now includes references to the Marimba and iFDO papers by default, so downstream users have the canonical citations to hand. Pipelines can prepend their own related material or override a default by reusing its URI.
- **EXIF-Preserving Image Transforms**: The image-processing library now preserves source EXIF and ICC profiles through both the PIL transforms and the OpenCV filters (CLAHE, Gaussian blur, sharpen), with the orientation tag reset only when an image is actually rotated or flipped. Metadata is no longer silently dropped when images are processed.
- **iFDO Completeness Check**: Packaging now reports which schema-required iFDO fields have been left unpopulated, both in the log and as an end-of-run summary panel. This is a non-fatal completeness signal: it never fails packaging or changes the output, it just makes gaps visible so they can be addressed.
- **iFDO v2.2.1 Output**: Generated iFDO files now target schema version v2.2.1 (previously v2.1.0), including normalising `image-entropy` to the [0, 1] range the schema expects.
- **Resilient Map Rendering**: Dataset map generation is now best-effort. On hosts without outbound internet access (for example, HPC compute nodes), Marimba logs a warning and skips `map.png` rather than aborting the entire packaging run, and uses a request timeout and an identifiable User-Agent so tile fetches fail fast and stay within OpenStreetMap's usage policy.
- **Quality-of-Life Improvements**: Dataset metadata now serialises as JSON by default, which is dramatically faster than YAML on large datasets; process pools pin an explicit `forkserver` start method for consistent, deadlock-free concurrency across Python versions; FFmpeg is no longer a required system dependency (video summaries and corruption checks now run through PyAV); several iFDO deduplication edge cases on small datasets are fixed; and the iFDO record embedded in each image's EXIF now uses the same configured datetime format as `ifdo.json`.

## Notable Changes

This release contains no hard API removals or incompatible CLI changes, but the following behaviour and output changes are worth knowing when upgrading from v1.1.0:

- **Deterministic UUIDs**: Re-packaging the same input now yields the same `image-set-uuid` and per-image `image-uuid` values, where v1.1.0 generated random ones. A dataset packaged with v1.2.0 will therefore have different identifiers than the same input packaged with v1.1.0.
- **iFDO output version**: Generated iFDO files are now v2.2.1 instead of v2.1.0.
- **Metadata default format**: Dataset metadata (iFDO and generic) is now written as JSON by default instead of YAML. Pass `--metadata-output yaml` to the `package` command to keep the previous YAML output.
- **Manifest header lines**: `manifest.txt` now begins with `# image-set-uuid`, `# image-set-name`, and `# hash-algorithm` comment lines. Third-party parsers that expect only `path:hash` lines should skip lines beginning with `#`.
- **FFmpeg no longer required**: ExifTool remains the only required external system binary; FFmpeg is no longer needed.
- **Pipeline author note**: `BaseMetadata.create_dataset_metadata` now takes a required `logger` argument. Pipelines that subclass `BaseMetadata` and override this method must add the parameter to their signature.
